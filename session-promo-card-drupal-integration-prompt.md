## Goal

Add a **Session Promo Card** feature to the Session Dashboard page with two interaction modes:

### Mode 1 — Single Card Preview + Download
From any session row in the Session Dashboard, a user can:
- Click a **"Promo Card"** action link in the row's ACTIONS column
- A **Drupal off-canvas drawer** (or AJAX dialog) opens showing the rendered promo card at a legible preview size
- The drawer has two export buttons: **Download PNG** and **Download PDF**
- Both exports produce a full 1800 × 1200 px (6×4 in @ 300 DPI) output matching the card layout

### Mode 2 — Bulk Download
From the Session Dashboard, a user can:
- Select multiple session rows using the existing checkboxes
- Choose **"Download Promo Cards (ZIP)"** from the bulk Actions dropdown
- A **Drupal Batch API** job runs — one PNG per selected session, packaged into a single ZIP
- A progress bar is shown during processing
- On completion, the ZIP is offered for download

## Architecture Decisions — Enforce These

### A. New render variant: `session_promo_card`

Add `session_promo_card` as a new variant in `html_to_image_renderer`:

- Register theme hook in `html_to_image_renderer.module`:
  ```php
  'html_to_image_renderer_variant_session_promo_card' => ['data' => NULL]
  ```
- Add to `RenderAssetController::VARIANT_MAP`:
  ```php
  'session-promo-card' => 'variant_session_promo_card'
  ```
- Add to `RenderController::DATA_VARIANTS`.
- Route path: `/render/asset/session-promo-card/{nid}`
- Template: `html-to-image-renderer--variant-session-promo-card.html.twig`
  - Primary location: `web/modules/custom/html_to_image_renderer/templates/`
  - Theme override: `web/themes/custom/somebody_theme/templates/marketing/`
- This template receives a flat `#data` array — **never a Drupal entity**. No Twig functions that call services.

### B. Extend `SessionDataNormalizer` for the new variant

`SessionDataNormalizer::normalize()` currently does not return:
- `series_label` — from `field_header_title` or the Spotlight program term
- `times_by_zone` — ET/CT/MT/PT breakdown from the session timestamp
- `short_url` — the shortened registration URL
- `qr_base64` — base64 QR code PNG (already generated by `SpotlightMicrositeApiService`)

**Do not duplicate logic from `SpotlightMicrositeApiService`.** Instead:

1. Check whether `SpotlightMicrositeApiService::searchSessions()` or a related service method already resolves these fields.
2. If yes — inject `SpotlightMicrositeApiService` (or extract a shared helper service) into `SessionDataNormalizer` and delegate.
3. If not — implement a new private method `buildTimezoneRow(int $timestamp): array` in `SessionDataNormalizer` that returns `['ET' => '6:00 PM ET', 'CT' => '5:00 PM CT', 'MT' => '4:00 PM MT', 'PT' => '3:00 PM PT']`.
4. Add `series_label` resolution: prefer `field_header_title` value; fallback to the session type term label; final fallback to `''`.
5. Add `short_url` to the normalizer output when available.
6. Add `qr_base64` — call existing QR generation logic; must not timeout the page render. Use an empty string fallback, not an exception.

The normalizer must remain entity-free in its output. All new fields go into the flat array it returns.

### C. Twig template — exact layout reproduction

The template must reproduce `facebook-promo-v2.html` layout exactly. Strip out:
- All JavaScript (search panel, debounce, export buttons, dom-to-image, jsPDF)
- Size-toggle controls
- The controls bar

Keep:
- All CSS custom properties and zone styles (copy the CSS block verbatim, inline in `<style>`)
- Zone 1: Logo bar — AF logo (left) | date/time block (center) | STTT logo (right)
- Zone 2: Session band — series label (gold italic) + session title (white bold)
- Zone 3: 3-column presenter section — headshot + name + cred | date pill + description + register URL | QR code + "SCAN TO REGISTER"
- Zone 4: Footer disclaimer

**Logos must be embedded as base64** in the template itself. Extract the AF logo and STTT_Logo_Basic base64 strings from `facebook-promo-v2.html`. Do not load them via HTTP in the Twig template — `RenderAssetController` returns a bare Response with no Drupal page chrome, so asset pipeline is unavailable.

Date formatting in Twig (replicate the JS logic from `populateCard()`):
```twig
{# Header date — "Thursday, May 21st" #}
{{ data.date | date('l, F j') }}{{ data.date | date('j') | ordinal_suffix }}

{# Header time — "@ 6pm ET" #}
@ {{ data.times_by_zone.ET | replace({':00 ': ' '}) | lower }}
```

If `data.series_label` is empty, render a non-breaking space (`&nbsp;`) in `.band-series-label` so the band height does not collapse.

If `data.presenter.image_url` is empty (no headshot), render a placeholder circle filled with `var(--rose)` and the presenter's initials.

### D. Puppeteer PNG capture for exports

The existing Puppeteer/ContentAssetGenerator pipeline already captures `/render/asset/{variant}/{nid}` as PNG. Extend it:

- Add `TYPE_SESSION_PROMO_CARD = 'session_promo_card'` constant to `ContentAssetGenerator`
- Set dimensions: `1800 × 1200` (matches 6×4 in @ 300 DPI)
- Set variant: `'session-promo-card'`
- Puppeteer viewport: `{ width: 1800, height: 1200, deviceScaleFactor: 1 }`

For **on-demand single-card export** (the drawer download buttons), do NOT queue — call Puppeteer synchronously via a new controller action with a reasonable timeout (15s). Return the PNG as a `BinaryFileResponse` with `Content-Disposition: attachment`.

For **bulk export**, use the Batch API (see Section F).

### E. Single card — AJAX drawer integration

Modify the Session Dashboard view to add a "Promo Card" action link per row:

1. Find the view's ACTIONS field (the column with pencil/trash icons visible in the screenshot).
2. Add a new link field or extend the existing dropbutton: label **"Promo Card"**, URL pattern `/session-dashboard/promo-card-preview/{nid}` (new route, see below).
3. The link must have `data-dialog-type="dialog"` and `data-dialog-options='{"width":960}'` attributes so Drupal's AJAX dialog system opens it in a modal/off-canvas drawer without page reload. Use `use-ajax` class.
4. New route: `sttt_session.promo_card_preview`
   - Path: `/session-dashboard/promo-card-preview/{node}`
   - Controller: `PromoCardPreviewController::preview(NodeInterface $node)`
   - `_permission: 'access session dashboard'` (use whatever permission guards the dashboard)
   - Returns a render array containing:
     - An `<iframe>` pointing to `/render/asset/session-promo-card/{nid}` scaled to fit the dialog width (use `transform: scale()` CSS trick to scale 1800px card to ~900px display width)
     - Two download links: **PNG** → `/session-dashboard/promo-card-download/png/{nid}`, **PDF** → `/session-dashboard/promo-card-download/pdf/{nid}`
5. New routes for downloads:
   - `sttt_session.promo_card_download_png`: `PromoCardDownloadController::downloadPng(NodeInterface $node)` — triggers Puppeteer synchronously, returns PNG
   - `sttt_session.promo_card_download_pdf`: `PromoCardDownloadController::downloadPdf(NodeInterface $node)` — generates PNG via Puppeteer, embeds in jsPDF server-side equivalent (use `TCPDF` or `Dompdf` if available in composer; otherwise embed PNG into PDF via `imagick` if installed; document the dependency requirement)

**DI requirements:**
- `PromoCardPreviewController` injects: `entity_type.manager`, `url_generator`
- `PromoCardDownloadController` injects: `entity_type.manager`, `sttt_session.content_asset_generator` (or equivalent Puppeteer service)

### F. Bulk download — VBO action + Batch API

1. Check whether the Session Dashboard view already uses `views_bulk_operations` (VBO) or a custom action form. Read the view config before deciding.
2. If VBO is already installed and configured — add a new VBO action plugin:
   - Plugin ID: `sttt_download_promo_cards`
   - Label: `Download Promo Cards (ZIP)`
   - Action type: `\Drupal\views_bulk_operations\Action\ViewsBulkOperationsActionBase`
   - `executeMultiple()` passes NIDs to a Batch API process function
3. If no VBO — add a custom action via `hook_form_alter` on the view's exposed form, following the same pattern as existing bulk actions in the dashboard.

**Batch process (`PromoCardBatchBuilder`):**
```php
// Batch definition
$batch = [
  'title' => t('Generating promo cards…'),
  'operations' => array_map(fn($nid) => [
    [PromoCardBatchBuilder::class, 'processItem'],
    [$nid, $tempDir]
  ], $nids),
  'finished' => [PromoCardBatchBuilder::class, 'finished'],
];
```

Each `processItem($nid, $tempDir, &$context)`:
- Calls Puppeteer via `ContentAssetGenerator` (or equivalent) for the `session_promo_card` variant
- Saves PNG to `$tempDir/{nid}-session-promo-card.png`
- Updates `$context['results']` with file path

`finished($success, $results, $operations)`:
- Packages all PNGs into a ZIP using PHP's `ZipArchive`
- Stores ZIP in `private://` filesystem (not `public://` — avoid exposing individual card files)
- Stores the ZIP path in `$_SESSION` or a temp store keyed by current user UID
- Redirects to a download route: `/session-dashboard/promo-card-bulk-download`

Bulk download route (`sttt_session.promo_card_bulk_download`):
- Controller reads path from temp store
- Returns `BinaryFileResponse` with `Content-Disposition: attachment; filename="promo-cards-{date}.zip"`
- Deletes the temp ZIP after streaming (use `deleteFileAfterSend: true` if Symfony version supports it, otherwise register a shutdown function)

```
  └── "Promo Card" link (use-ajax)
        └── AJAX Dialog opens PromoCardPreviewController
              └── iframe → /render/asset/session-promo-card/{nid}

              └── Download PNG → PromoCardDownloadController::downloadPng()
                    └── Puppeteer → 1800×1200 PNG → BinaryFileResponse
              └── Download PDF → PromoCardDownloadController::downloadPdf()
                    └── Puppeteer PNG → embed in PDF → BinaryFileResponse

Session Dashboard checkboxes → Bulk action → "Download Promo Cards (ZIP)"
  └── PromoCardBatchBuilder::processItem() × N (Batch API)
        └── Puppeteer PNG per session → private:// temp dir
  └── PromoCardBatchBuilder::finished()
        └── ZipArchive → private:// ZIP → temp store
  └── /session-dashboard/promo-card-bulk-download
        └── BinaryFileResponse → ZIP → delete temp file
```

## Coding Standards — Non-Negotiable

- Drupal 10 coding standards. Run `phpcs --standard=Drupal` before submitting. Zero warnings.
- Dependency injection throughout. No `\Drupal::` static calls inside class methods.
- All new routes must have explicit `_permission` checks. No `_access: 'TRUE'`.
- `SessionDataNormalizer` output remains a flat array of scalars — no entity objects, no service proxies.
- Twig template must be entity-free. No `entity` functions, no `path()` calls that require a route resolver at render time (the template is captured by Puppeteer via a bare HTTP response).
- Batch callbacks must be static methods (Drupal Batch API requirement).
- `ZipArchive` usage must check `open()` return value; handle failure gracefully with a user-facing error message.
- Use `private://` for all temporary ZIP files. Never write to `public://`.
- `BinaryFileResponse` streaming must set correct MIME type: `image/png` for PNG, `application/zip` for ZIP.

## Risk Notes

1. **Puppeteer synchronous timeout on single-card download**: 15s is the cap. If the session has no headshot (`headshot_base64` empty), the card renders faster. If Puppeteer is unavailable (queue worker down), the controller must return a 503 with a human-readable message — not a white screen.
2. **QR base64 generation in normalizer**: The QR library (`chillerlan/php-qrcode`) is a Composer dependency in the microsite repo, not necessarily in the Drupal repo. Check `composer.json` first. If not present, add it. Do not silently swallow the missing-dependency error.
3. **Bulk ZIP with many sessions**: A 37-session dashboard (visible in the screenshot) generating 37 × Puppeteer calls is a worst-case batch. Each call is ~3–8s. Worst-case: ~5 minutes. The Batch API progress bar handles this correctly — do not try to do it in a single request.
4. **Base64 logos in Twig**: The AF logo base64 string is ~16KB; STTT_Logo_Basic is ~135KB. Total template size will be large but acceptable for server-side rendering. Do not lazy-load from external URLs — the Puppeteer capture environment may not have internet access.
5. **AJAX dialog width vs card width**: The card is 1800px. The dialog will be 960px. Use `transform: scale(0.5); transform-origin: top left;` on the iframe's content wrapper, and size the iframe element at 900px wide × 600px tall. The card inside renders at true 1800×1200 but displays at half scale. This is intentional — it is a preview, not the export.
6. **VBO dependency**: If `views_bulk_operations` is not in `composer.json`, implement a custom form alter action instead. Do not add VBO as a new dependency without confirming with Sukh first.
